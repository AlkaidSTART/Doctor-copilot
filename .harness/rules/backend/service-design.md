# 服务设计规则

## 服务架构

### 架构风格
- 使用 Monorepo 结构
- 采用分层架构（UI、API、Service、Repository）
- 使用 Server Actions 作为写操作入口

### 模块划分
- 按功能域划分模块（patient、care-plan、task、timeline）
- 每个模块包含组件、actions、queries、schemas
- 模块间通过 API 或事件通信

### 依赖管理
- 模块间依赖单向流动
- 避免循环依赖
- 使用共享类型和工具函数

## 服务设计原则

### 单一职责
- 每个服务只负责一个业务领域
- 避免服务过度膨胀
- 服务代码不超过 500 行

### 高内聚低耦合
- 相关功能聚集在一起
- 模块间接口简洁
- 降低模块间依赖

### 可扩展性
- 支持水平扩展
- 支持功能扩展
- 使用插件化设计

### 可测试性
- 服务代码可独立测试
- 使用依赖注入
- 避免静态依赖

## 服务开发规范

### 文件结构
```
features/
  patient/
    components/      # UI 组件
    actions/         # Server Actions
    queries/         # 数据查询
    schemas/         # 数据验证
    types/           # 类型定义
    hooks/           # 自定义 hooks
```

### Server Actions
- 使用 `'use server'` 指令
- 作为写操作的唯一入口
- 包含权限检查和参数验证
- 返回标准响应格式

### 数据访问
- 使用 Supabase 进行数据访问
- 使用 RLS 进行行级权限控制
- 使用 RPC 进行复杂查询

### 业务逻辑
- 业务逻辑放在服务层
- 避免在 UI 层处理业务逻辑
- 业务规则可配置化

## 错误处理

### 错误分层
- 业务错误：业务规则验证失败
- 系统错误：数据库、网络等基础设施错误
- 认证错误：权限不足、未认证

### 错误传播
- 使用 try-catch 捕获错误
- 统一错误处理中间件
- 返回标准错误格式

### 错误日志
- 记录错误详情
- 包含请求上下文（request_id、user_id）
- 使用结构化日志

## 事务管理

### 事务边界
- 使用 Supabase RPC 处理多表操作
- 使用数据库事务保证数据一致性
- 避免跨服务事务

### 分布式事务
- 使用最终一致性
- 使用事件驱动架构
- 使用 Outbox 模式

## 安全规范

### 权限控制
- 使用 RBAC 进行角色权限控制
- 使用 RLS 进行行级权限控制
- 最小权限原则

### 输入验证
- 使用 Zod Schema 验证输入
- 防止注入攻击
- 防止恶意输入

### 数据保护
- 敏感数据加密存储
- 敏感数据脱敏输出
- 审计日志记录

## 性能优化

### 缓存策略
- 使用 Redis 缓存热点数据
- 使用 HTTP 缓存
- 支持缓存失效机制

### 数据库优化
- 使用索引优化查询
- 使用连接池管理连接
- 避免 N+1 查询

### 异步处理
- 使用消息队列处理异步任务
- 使用 Server Actions 进行异步操作
- 使用 WebSocket 进行实时通信

## 测试规范

### 测试覆盖
- 单元测试：测试服务逻辑
- 集成测试：测试模块间交互
- E2E 测试：测试端到端流程

### 测试工具
- 使用 Jest 进行单元测试
- 使用 Supabase Test Helpers 进行集成测试
- 使用 Cypress 进行 E2E 测试

### 测试数据
- 使用测试数据库
- 使用工厂模式生成测试数据
- 测试后清理数据
