# 架构决策记录

## ADR 001 - Monorepo 结构

### 决策背景
项目需要支持多应用和共享包的开发模式。

### 决策内容
采用 Monorepo 结构，使用 pnpm workspace 管理项目。

### 目录结构
```
apps/
  web/                    # Next.js Web 应用

packages/
  ui/                     # UI 组件库
  config/                 # 配置共享
  types/                  # 类型定义
  utils/                  # 工具函数
```

### 决策理由
- 代码复用：共享组件、类型、工具函数
- 依赖管理：统一管理依赖版本
- 构建效率：共享构建配置
- 开发体验：单一仓库管理

### 影响
- 需要配置 pnpm workspace
- 需要配置 TypeScript 路径别名
- 需要配置构建脚本

## ADR 002 - Server Actions 作为写操作入口

### 决策背景
需要统一后端写操作的入口。

### 决策内容
使用 Next.js Server Actions 作为所有写操作的入口。

### 决策理由
- 简化后端逻辑
- 自动 CSRF 保护
- 更好的类型安全
- 与前端无缝集成

### 影响
- 需要学习 Server Actions 使用方式
- 需要配置 CORS
- 需要处理错误边界

---

## ADR 003 - RLS 行级安全性

### 决策背景
需要实现细粒度的数据访问控制。

### 决策内容
在 Supabase 中启用 RLS，使用 org_id + role_claim 作为策略核心。

### 决策理由
- 数据隔离：不同机构数据隔离
- 权限控制：基于角色的访问控制
- 安全性：最小权限原则
- 审计：可追踪的数据访问

### 影响
- 需要设计 RLS 策略
- 需要测试 RLS 规则
- 需要处理权限不足的情况

## ADR 004 - AI 多模型路由

### 决策背景
需要支持多个 AI 模型，根据任务类型选择合适的模型。

### 决策内容
实现 Model Router，根据任务类型、延迟、成本选择合适的模型。

### 决策理由
- 灵活性：支持多种模型
- 成本优化：选择性价比最高的模型
- 性能优化：选择响应最快的模型
- 可靠性：多模型回退

### 影响
- 需要设计路由逻辑
- 需要评估模型性能
- 需要监控模型使用情况

## ADR 006 - Feature 模块化结构

### 决策背景
需要按功能域组织代码，提高可维护性。

### 决策内容
采用 Feature 模块化结构，每个功能域包含组件、actions、queries、schemas。

### 目录结构
```
features/
  patient/
    components/           # UI 组件
    actions/              # Server Actions
    queries/              # 数据查询
    schemas/              # 数据验证
    types/                # 类型定义
    hooks/                # 自定义 hooks
```

### 决策理由
- 职责清晰：每个模块只负责一个功能域
- 可维护性：代码组织清晰
- 可测试性：模块可独立测试
- 可扩展性：易于添加新功能

### 影响
- 需要设计模块边界
- 需要定义模块接口
- 需要处理模块间依赖
